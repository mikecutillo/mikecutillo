## Known bugs (next session)

### 1. LinkedIn title/location extraction returns blank (short-term cosmetic only)
**Symptom:** Sending an Anaplan job from a LinkedIn search page created a
pipeline card with `company: "linkedin.com"` and `title: "Untitled Role"`
instead of the real values. URL was preserved correctly so the job is still
identifiable.

**Root cause (hypothesis):** Chrome doesn't inject content scripts into
tabs that were already open when the extension was installed. The popup's
`tabMsg('getJobPreview')` returned no-response, fell through to the tab-metadata
fallback, where `chrome.tabs.query` returned a tab with `title === undefined`
(LinkedIn SPAs dynamically overwrite `document.title` and `activeTab`
permission doesn't always capture it). Company came from the hostname fallback.

**Fixes for next session:**
1. Immediate: reload any pre-existing tab after installing the extension.
   Newly-loaded tabs get the content script automatically.
2. Harden extraction in `apply-assistant-extension/content/send-to-pipeline.js`
   and `popup.js`:
   - Accept `og:title` alone (don't gate on `og:description` being present too).
   - Popup fallback should call `chrome.scripting.executeScript` to read
     `document.title` directly from the page rather than `tab.title`.
   - LinkedIn selectors are stale — swap to data attributes or the newer
     `.job-details-jobs-unified-top-card__*` family (need to inspect current
     LinkedIn DOM to get reliable selectors).
   - Consider a last-ditch heuristic: parse the job ID out of
     `?currentJobId=NNNNNNN` in the URL and fetch `https://www.linkedin.com/jobs/view/NNNNNNN/`
     via the extension to grab JSON-LD there (that route is usually schema.org-rich).

### 2. Potentially: other boards untested in the live extension flow
Only the LinkedIn search page was tested live. Workday/Greenhouse/Lever/Ashby/iCIMS
send-to-pipeline paths use site-specific selectors in send-to-pipeline.js that
haven't been exercised against real pages.

## How the extension is wired
- Extension ID in Chrome: visible on `chrome://extensions` card.
- Service worker logs: Details → "Inspect views: service worker".
- All extension → Mission Control traffic goes through `background.js` which
  auto-detects between `localhost:3333` and `localhost:52323` on the first
  request and caches the base for 30s.
- Extension handlers: `ping`, `sendToPipeline`, `matchFields`, `saveAnswer`,
  `logUnknown`, `getCredential`.
